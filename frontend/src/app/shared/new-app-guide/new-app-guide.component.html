1

Scaffold — new-app.sh

Depuis la racine dev/ :

bash scripts/new-app.sh

Le script demande, dans l'ordre :

  • Nom de l'application (minuscules + tirets, ex : mon-app)
  • Type : Spring Boot seul / Spring+Angular / Django seul / Django+Angular / Angular seul
  • Port backend (suggéré automatiquement, ≥ 8083)
  • Port frontend (suggéré automatiquement, ≥ 4200)
  • Scaffold via Docker (répondre O)
  • Groupe requis pour le cloisonnement (voir étape 3)
  • Pour une app avec base : instance PostgreSQL — postgres (partagée, défaut) ou postgis (SIG uniquement)
Il crée dev/<app>/ complet (backend, frontend, docker-compose, Dockerfiles, nginx, .env…), ajoute l'app au .gitignore du dépôt parent, enregistre les ports dans .ports, ajoute le schéma dans infra/init/00_schemas.sql et crée .keycloak-client-opts. Il ne crée ni le dépôt GitHub ni le client Keycloak.
2

Dépôt GitHub + sous-module

L'app devient un sous-module git pointant vers son propre dépôt Sacha37420/<app> :

cd mon-app
git init && git checkout -b main
git add . && git commit -m "feat: initial scaffold"
gh repo create Sacha37420/mon-app --public
git remote add origin https://github.com/Sacha37420/mon-app.git
git push -u origin main
cd ..

# Retirer du .gitignore (new-app.sh l'y avait ajouté) et enregistrer comme sous-module
sed -i '/^mon-app\/$/d' .gitignore
git submodule add https://github.com/Sacha37420/mon-app.git mon-app
3

Cloisonner l'app (à ne pas sauter)

Le lab est exposé sur Internet. Être authentifié dans le realm ssolab ne doit donner accès à rien : toute app se réserve à un ou plusieurs groupes LDAP via --require-group dans <app>/.keycloak-client-opts : 

--public --port 4208 --caddy-path mon-app --require-group famille,amis
Important : un nouvel inscrit n'a aucun groupe ⇒ accès à aucune app. C'est voulu. Laisser --require-group vide rend l'app accessible à tout compte du realm — new-app.sh le signale bruyamment si on le fait.

Compléter aussi mon-app/.env (généré par le scaffold) : SECRET_KEY, DEBUG=True en dev, DOMAIN=CHANGE_ME tant que le lab reste en HTTP local.

4

Déploiement complet — setup2.sh

bash scripts/setup2.sh mon-app --yes

Ce script enchaîne, dans l'ordre :

  • clean2.sh <app> — arrête et supprime les containers de l'app
  • reset_url.sh — propage LAN/WAN/Keycloak dans tous les .env
  • Démarrage de sso-lab (Keycloak + LDAP + Caddy)
  • Attente que Keycloak réponde (jusqu'à 300 s)
  • create-app-client.sh <app> — crée/met à jour le client Keycloak (secret, redirect URIs, claim groups, rôle <client>-access, flow require-<client>)
  • recompose_docker.sh --app <app> --force — build et démarre les containers
  • get-ports-list.sh — régénère ports.env
  • open-bbox-ports2.sh — ouvre les ports sur le routeur Bbox si accessible
create-app-client.sh peut aussi être appelé seul pour recréer/mettre à jour un client sans tout redéployer : 
bash scripts/create-app-client.sh mon-app $(cat mon-app/.keycloak-client-opts)
Après tout changement de cloisonnement, tester dans les deux sens : un membre du groupe passe, un non-membre (avec ou sans session SSO active) est refusé.
